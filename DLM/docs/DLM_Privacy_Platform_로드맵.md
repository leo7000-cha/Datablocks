# DLM → 개인정보 All-in-One 솔루션 로드맵

> 작성일: 2026-04-04
> "DB 안의 개인정보를 가장 잘 아는 솔루션"

---

## 1. 현재 DLM이 가진 핵심 역량

| 역량 | 활용 방향 |
|------|-----------|
| 멀티DB 지원 (Oracle, Tibero, MySQL, MariaDB, PG, MSSQL, DB2) | 어떤 DB든 개인정보 스캔/처리 가능 |
| DmlExecutor 대규모 배치 처리 | 대용량 개인정보 일괄 비식별화/파기 |
| 데이터 이동/변환 파이프라인 | 개인정보 탐지→분류→처리 자동화 |
| JobScheduler 스케줄링 | 정기 스캔/파기/보고서 자동 실행 |
| ConnectionProvider + HikariCP 풀링 | 다중 시스템 동시 접속 관리 |

## 2. 시장 분석 요약

### 2.1 시장 규모

- 2025년 국내 정보보호산업: **18.6조원** (전년 대비 10.5% 성장)
- 정보보호 기업 수: 1,780개사 (전년 대비 4.2% 증가)
- 개인정보 접속기록 관리 시장: **200억원+** (접속기록 보관 5년 확대로 급성장)

### 2.2 솔루션 유형별 사용률

| 솔루션 유형 | 사용률 |
|---|---|
| 정보유출방지(DLP) | 52.8% |
| 개인정보 암호화 | 51.9% |
| 개인정보 접속기록관리 | 40.2% |
| DRM(디지털 저작권 관리) | 22.0% |
| 개인정보 비식별화 | 19.2% |

## 3. 시장 트렌드 (2025-2026)

### 3.1 AI 기반 개인정보 탐지
- 트랜스포머/LLM 기반 탐지 (파수 AI레이더, 스파이스웨어 GPT-3)
- AI OCR로 이미지 내 개인정보 검출
- AI 문맥 기반 결합 개인정보 식별
- 2026년 인공지능 기본법 시행 → AI 기반 보안 솔루션 자체도 규제 대상

### 3.2 클라우드 환경 대응
- SaaS형 보안 솔루션 확산 (구독형)
- 제로 트러스트 모델 표준화
- 국가망 보안체계: 망분리 → 클라우드 보안체계 전환

### 3.3 데이터3법 / 마이데이터 확대
- '가명정보' 개념 도입 → 비식별화 솔루션 수요 폭증
- 마이데이터 확대 로드맵: 2025년(5개 부문) → 2026년(10개) → 2027년(전 분야)
- 동의관리, 접근제어, 접속기록관리, 암호화 수요 동시 견인

### 3.4 All-in-One 통합 플랫폼 가속화
- 개별 솔루션 관리 비용/보안 사각지대 문제 → 통합 플랫폼 전환
- 검출→분류→보호→모니터링→파기 전 라이프사이클 자동화
- DSPM (Data Security Posture Management) 개념 부상

## 4. 제품 비전: DLM Privacy Platform

> "DB 안의 개인정보를, 찾고 → 분류하고 → 보호하고 → 추적하고 → 파기하는" 올인원 플랫폼

```
┌──────────────────────────────────────────────────────────────────┐
│                    DLM Privacy Platform                          │
│                                                                  │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────────────┐ │
│  │ Privacy  │  │ Privacy  │  │ Privacy  │  │ Privacy          │ │
│  │ Scanner  │→ │ Shield   │→ │ Monitor  │→ │ Lifecycle Mgr    │ │
│  │          │  │          │  │          │  │                  │ │
│  │ 탐지/분류 │  │비식별화   │  │접속기록   │  │파기/동의/보고서   │ │
│  │          │  │/마스킹    │  │/이상탐지  │  │                  │ │
│  └──────────┘  └──────────┘  └──────────┘  └──────────────────┘ │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────────┐│
│  │              DLM Core Engine (기존 자산)                      ││
│  │   멀티DB 커넥션 | 배치 처리 | 스케줄링 | 데이터 이동/변환     ││
│  └──────────────────────────────────────────────────────────────┘│
└──────────────────────────────────────────────────────────────────┘
```

**핵심 기회**: 이지서티 같은 기존 강자들도 4개 제품을 묶어 All-in-One이라 하지만, **DB 레벨의 데이터 라이프사이클 관리**를 진짜로 통합한 솔루션은 없음. DLM의 강점이 바로 여기.

## 5. 기술 아키텍처: 하이브리드 (Java + Python)

### 5.1 왜 하이브리드인가

| | DLM에 붙이기 (Java Only) | 별도 개발 (Python Only) | **하이브리드 (추천)** |
|---|---|---|---|
| 기존 자산 재사용 | 100% | 0% (재구현) | **핵심만 재사용** |
| AI/ML 생태계 | 빈약 | **최강** (spaCy, transformers, pandas) | **Python으로 분리** |
| 멀티DB 처리 | **이미 완성** | SQLAlchemy로 재구현 | **DLM 엔진 그대로** |
| 운영 복잡도 | 단순 | 단순 | 약간 증가 |
| 확장성 | 모놀리식 심화 | 좋음 | **독립 스케일링** |

> 기존 DLM은 **"데이터를 다루는 근육"**, Python 서비스는 **"개인정보를 판별하는 두뇌"** 역할

### 5.2 시스템 구성도

```
┌─────────────────────────────────────────────────────────┐
│                   Web UI (React/Vue)                     │
└──────────────────────┬──────────────────────────────────┘
                       │
        ┌──────────────┴──────────────┐
        ▼                             ▼
┌───────────────────┐     ┌────────────────────────┐
│  DLM Core (Java)  │     │  Privacy AI (Python)   │
│  Spring Boot      │◄───►│  FastAPI               │
│                   │ REST│                        │
│ • 멀티DB 커넥션    │ API │ • 개인정보 패턴 탐지    │
│ • 배치 처리        │     │ • NLP 기반 비정형 탐지  │
│ • 스케줄링         │     │ • OCR (이미지 내 검출)  │
│ • 비식별화 실행     │     │ • 이상행위 ML 탐지     │
│ • 접속기록 수집     │     │ • 적정성 평가 통계     │
│ • 파기 실행        │     │                        │
└───────────────────┘     └────────────────────────┘
```

### 5.3 역할 분담

**DLM Java가 잘하는 것** → 그대로 유지/확장
- DB 직접 연결해서 데이터 읽고/쓰고/삭제 (이미 7개 DB 지원)
- 대용량 배치 처리 (DmlExecutor 파티션 병렬)
- 스케줄링 (JobScheduler)
- 트랜잭션 관리 (커밋/롤백/세이브포인트)

**Python이 압도적인 것** → 별도 서비스 (FastAPI)
- `spaCy` / `transformers`: 이름, 주소 같은 비정형 개인정보 NLP 탐지
- `pytesseract` / `easyocr`: 이미지 내 개인정보 OCR
- `scikit-learn`: 이상행위 탐지 ML 모델
- `pandas` + `ARX`: k-익명성, l-다양성 적정성 평가
- 정규식 탐지 엔진도 Python이 훨씬 개발 생산성 높음

### 5.4 Phase별 Java/Python 분담

| Phase | DLM Java (확장) | Python (신규) |
|-------|-----------------|---------------|
| **1. Scanner** | DB 메타 조회, 샘플링 쿼리 실행 | 패턴 매칭, 개인정보 분류 판정 |
| **2. Shield** | 배치 UPDATE/INSERT 실행 | 비식별 알고리즘, 적정성 평가 |
| **3. Monitor** | 접속 로그 수집/저장 | 이상행위 탐지 ML |
| **4. Lifecycle** | 파기 실행, 분리보관 | 보고서 생성 |
| **5. AI 강화** | - | 전면 AI 엔진 고도화 |

### 5.5 실제 흐름 예시 (Phase 1 Scanner)

```
1. DLM Java: SELECT column_name, data_type FROM information_schema.columns
             → 메타정보 + 샘플 데이터 100건 추출

2. REST API 호출 → Python Privacy AI 서비스

3. Python: spaCy NER + 정규식으로 개인정보 판정
           → {"column": "user_name", "type": "이름", "confidence": 0.95}

4. DLM Java: 결과를 privacy_scan_result 테이블에 저장
             → 대시보드에서 조회
```

## 6. Phase별 개발 계획

### Phase 1: Privacy Scanner (탐지/분류) — 3~4개월

**목표**: DB 내 개인정보 자동 탐지 및 분류

| 기능 | 상세 | 우선순위 |
|------|------|----------|
| **컬럼 메타 스캔** | 컬럼명 패턴으로 개인정보 후보 자동 식별 (name, phone, email, ssn, jumin 등) | P0 |
| **데이터 샘플링 검증** | 실제 데이터 샘플링 후 정규식+체크섬으로 개인정보 확인 (주민번호 검증, 전화번호 패턴 등) | P0 |
| **개인정보 분류 체계** | 고유식별정보, 민감정보, 일반개인정보 자동 분류 | P0 |
| **멀티DB 동시 스캔** | 기존 ConnectionProvider 활용, 여러 DB를 병렬 스캔 | P1 |
| **스캔 결과 대시보드** | DB별/테이블별/컬럼별 개인정보 현황 시각화 | P1 |
| **정기 스캔 스케줄** | JobScheduler 확장, 주/월 단위 자동 스캔 | P2 |

**DLM 자산 활용**: DmlExecutor의 멀티DB SELECT + SqlUtil의 메타 조회 + JobScheduler 스케줄링

**차별점 vs 경쟁사**: DB 레벨 직접 스캔 (웹/파일 레벨이 아닌 원천 데이터 스캔)

---

### Phase 2: Privacy Shield (비식별화/마스킹) — 4~5개월

**목표**: 탐지된 개인정보에 대한 비식별화 처리

| 기능 | 상세 | 우선순위 |
|------|------|----------|
| **실시간 마스킹** | SELECT 시 동적 마스킹 (View/Proxy 방식) | P0 |
| **배치 비식별화** | 대량 데이터 일괄 가명처리 (DmlExecutor 배치 활용) | P0 |
| **비식별 기법 라이브러리** | 마스킹, 일반화, 범주화, 총계처리, 데이터 삭제, 가명처리 | P0 |
| **가명처리 정책 관리** | 테이블/컬럼별 비식별 규칙 설정 및 관리 | P1 |
| **적정성 평가** | k-익명성, l-다양성, t-근접성 자동 검증 | P1 |
| **비식별 데이터 추출** | 안전한 가명정보 데이터셋 생성 (연구/분석용) | P2 |

**DLM 자산 활용**: DmlExecutor의 transformAndInsert 확장 → 변환 과정에 비식별화 로직 삽입

**차별점**: 이지서티 IDENTITY SHIELD가 2,500억건 처리를 강점으로 내세우는데, DLM의 파티션 병렬 처리 + HikariCP 풀링으로 대등한 성능 가능

---

### Phase 3: Privacy Monitor (접속기록/모니터링) — 4~5개월

**목표**: 개인정보 처리 이력 관리 및 이상행위 탐지

| 기능 | 상세 | 우선순위 |
|------|------|----------|
| **접속기록 수집** | DB 접속 로그 수집 (5년 보관 의무 대응) | P0 |
| **조회/다운로드 이력** | 개인정보 테이블 접근 이력 기록 | P0 |
| **위/변조 방지** | 로그 무결성 보장 (해시 체인 등) | P0 |
| **이상행위 탐지** | 비정상 대량 조회, 업무외 시간 접속, 패턴 이탈 감지 | P1 |
| **소명 프로세스** | 이상행위 탐지 시 소명 요청/응답 워크플로우 | P1 |
| **실시간 대시보드** | 접속 현황, 이상징후, 소명 현황 모니터링 | P2 |

**DLM 자산 활용**: 기존 InnerStep 패턴의 이력 추적 노하우 + 배치 로그 수집

**시장 기회**: 접속기록 보관 5년 확대로 시장 200억+ 규모, 금융/민간 확산 중

## 8. 경쟁 포지셔닝 전략

| vs 경쟁사 | DLM Privacy Platform 차별점 |
|-----------|---------------------------|
| **이지서티** | DB 원천 데이터 직접 처리 (이지서티는 웹/네트워크 레벨 필터링 중심) |
| **파수** | 멀티DB 네이티브 지원 (파수는 파일/문서 중심, DB는 제한적) |
| **소만사** | 에이전트리스 (소만사는 PC 에이전트 필수, 관리 부담) |
| **컴트루** | 실시간 배치 처리 성능 (DmlExecutor 병렬 처리 엔진) |
| **스파이스웨어** | 온프레미스+클라우드 하이브리드 (스파이스웨어는 SaaS only) |

**핵심 메시지**: *"DB 안의 개인정보를 가장 잘 아는 솔루션"* — 웹/파일/네트워크가 아닌 **데이터가 실제로 저장된 DB 레벨**에서의 올인원 보호
