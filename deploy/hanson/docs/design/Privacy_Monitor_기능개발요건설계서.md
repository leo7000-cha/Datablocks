# Privacy Monitor — 개인정보 접속기록관리 기능 개발요건 설계서

> 작성일: 2026-04-06
> 버전: v1.0
> 로드맵 위치: Phase 3 (Privacy Monitor)

> **용어 안내**: 본 문서에서 "DLM"은 X-One 플랫폼의 기술 내부 명칭(Data Lifecycle Management)이며, 코드·인프라·DB에서 사용되는 기술명이다. 대외적으로는 "X-One 통합 데이터 관리 플랫폼"으로 표기한다.

---

## 1. 개요

### 1.1 목적

개인정보처리시스템에 대한 접속기록을 수집·저장·분석하여 개인정보보호법 및 금융감독규정의 법적 요건을 충족하고, 이상행위를 탐지·소명하는 통합 접속기록관리 모듈을 개발한다.

### 1.2 법적 근거

| 법령/고시 | 조항 | 핵심 내용 |
|-----------|------|-----------|
| **개인정보 보호법** | 제29조 | 안전조치 의무 |
| **안전성 확보조치 기준** | 제8조 1항 | 접속기록 보관 (1년/2년) |
| **안전성 확보조치 기준** | 제8조 2항 | 월 1회 이상 점검, 다운로드 사유 확인 의무 |
| **안전성 확보조치 기준** | 제8조 3항 | 접속기록 위변조·도난·분실 방지 |
| **전자금융감독규정** | 제13조 3항 | 이용자정보 조회 기록 1년 보존 |
| **전자금융감독규정** | 제13조 4항 | 가동기록: 접속일시·접속자·접근내용 |
| **전자금융감독규정** | - | 이용자 중요원장 접근 기록 5년 보존 |
| **ISMS-P** | 2.9.4 / 2.9.5 | 로그 및 접속기록 관리·점검 |

### 1.3 접속기록 필수 항목 (5W1H)

| 항목 | 설명 | DB 컬럼 예시 |
|------|------|-------------|
| **Who** (식별자) | 개인정보취급자 계정 | `user_id`, `user_name`, `dept` |
| **When** (접속일시) | 접속 시점 | `access_time` |
| **Where** (접속지) | 접속 원점 IP | `client_ip` |
| **Whom** (정보주체) | 처리한 정보주체 식별 | `target_info`, `search_condition` |
| **What** (수행업무) | 수행한 작업 유형 | `action_type` (조회/수정/삭제/다운로드) |
| **How** (접근경로) | 접근 수단 | `access_channel` (WEB/WAS/DB직접/API) |

### 1.4 보관 기간 요건

| 구분 | 보관 기간 |
|------|-----------|
| 일반 개인정보처리자 | **1년** 이상 |
| 5만명 이상 정보주체 처리 | **2년** 이상 |
| 고유식별정보·민감정보 처리 | **2년** 이상 |
| 금융사 이용자 중요원장 | **5년** |

### 2.2 접속기록 수집 방식

**Phase 3-A에서는 DB 로그 파싱 방식**을 우선 구현한다.

| 방식 | 구현 시기 | 설명 |
|------|-----------|------|
| **DB Audit Log 파싱** | Phase 3-A (우선) | 대상 DB의 감사 로그를 주기적으로 수집·파싱 |
| **DLM 자체 로그** | Phase 3-A (우선) | DLM을 통한 비식별화/조회 작업의 자체 접속기록 |
| **WAS 에이전트** | Phase 3-B (확장) | Tomcat/WebLogic 등에 에이전트 설치하여 수집 |
| **네트워크 미러링** | Phase 3-C (향후) | 네트워크 단에서 DB 쿼리 패킷 미러링 |

#### DB별 Audit Log 수집

| DBMS | Audit 기능 | 수집 방법 |
|------|-----------|-----------|
| **Oracle** | Unified Audit / Fine-Grained Audit | `UNIFIED_AUDIT_TRAIL` 뷰 조회 |
| **MySQL/MariaDB** | General Log / Audit Plugin | `mysql.general_log` 또는 MariaDB Audit Plugin |
| **PostgreSQL** | pgAudit | `pg_log` 파싱 또는 pgaudit 확장 |
| **MSSQL** | SQL Server Audit | `sys.fn_get_audit_file()` |
| **Tibero** | Tibero Audit | `DBA_AUDIT_TRAIL` 뷰 |
| **DB2** | DB2 Audit | `db2audit` 유틸리티 |

### 2.3 DLM 기존 자산 활용

| DLM 자산 | 활용 방법 |
|----------|-----------|
| **ConnectionProvider** | 대상 DB 감사 로그 접속 (기존 멀티DB 커넥션 풀 그대로 사용) |
| **JobScheduler** | 접속기록 수집 배치 스케줄링 (1분/5분/1시간 주기) |
| **DmlExecutor** | 대량 로그 INSERT 배치 처리 |
| **Discovery 결과** | 어떤 테이블/컬럼이 개인정보인지 → 모니터링 대상 자동 등록 |
| **PIICODE 체계** | 접속기록에 개인정보 등급/유형 연계 |

### 3.2 M1. 수집 (Collection)

#### M1.1 DB Audit Log 수집

| 항목 | 설명 |
|------|------|
| **기능** | 대상 DB의 감사 로그를 주기적으로 수집하여 접속기록 테이블에 저장 |
| **수집 주기** | 설정 가능 (기본: 5분, 최소: 1분) |
| **수집 대상** | DLM Discovery에서 개인정보로 탐지된 테이블에 대한 SELECT/UPDATE/DELETE/INSERT |
| **파싱 항목** | 사용자 계정, 접속일시, 클라이언트 IP, SQL문, 대상 테이블/컬럼, 영향받은 행 수 |
| **중복 방지** | 마지막 수집 위치(offset) 기록, 중복 INSERT 방지 |
| **에러 처리** | 수집 실패 시 재시도 (3회), 실패 로그 기록, 관리자 알림 |

**수집 흐름:**
```
1. JobScheduler → 수집 배치 트리거 (5분 주기)
2. ConnectionProvider → 대상 DB 접속
3. 마지막 수집 위치(offset)부터 Audit Log 조회
4. 파싱하여 5W1H 구조로 변환
5. DLM DB의 접속기록 테이블에 배치 INSERT
6. 수집 위치 업데이트
7. 해시값 생성 (위변조 방지)
```

#### M1.2 DLM 자체 접속기록

| 항목 | 설명 |
|------|------|
| **기능** | DLM 내에서 발생하는 개인정보 처리 작업을 자동 기록 |
| **대상 작업** | 비식별화 실행, Discovery 스캔, 데이터 조회, 보고서 다운로드, 설정 변경 |
| **구현 방식** | Spring AOP Interceptor로 자동 기록 |
| **기록 항목** | 로그인 사용자, 작업 유형, 대상 DB/테이블, 접속 IP, 타임스탬프 |

#### M1.3 수집 대상 시스템 관리

| 항목 | 설명 |
|------|------|
| **기능** | 접속기록 수집 대상 DB/시스템을 등록·관리 |
| **연계** | DLM에 이미 등록된 DB 커넥션 정보를 활용 |
| **설정** | DB별 수집 주기, 수집 방식(Audit Log/General Log), 대상 테이블 필터 |
| **상태 관리** | 수집 에이전트 상태(정상/오류/중지), 마지막 수집 시간, 누적 건수 |

### 3.3 M2. 저장/관리 (Storage)

#### M2.1 접속기록 저장

**핵심 테이블: `TBL_ACCESS_LOG`**

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `log_id` | BIGINT (PK, AUTO_INCREMENT) | 로그 고유 ID |
| `source_system_id` | VARCHAR(50) | 수집 원본 시스템 ID |
| `user_account` | VARCHAR(100) | 접속자 계정 (Who) |
| `user_name` | VARCHAR(100) | 접속자 이름 |
| `department` | VARCHAR(100) | 소속 부서 |
| `access_time` | DATETIME(3) | 접속일시 (When) |
| `client_ip` | VARCHAR(45) | 접속지 IP (Where) — IPv6 대응 |
| `action_type` | VARCHAR(20) | 수행업무 (What): SELECT/UPDATE/DELETE/INSERT/DOWNLOAD/EXPORT |
| `target_db` | VARCHAR(100) | 대상 DB명 |
| `target_schema` | VARCHAR(100) | 대상 스키마 |
| `target_table` | VARCHAR(200) | 대상 테이블 |
| `target_columns` | TEXT | 접근한 컬럼 목록 (개인정보 컬럼 포함 여부) |
| `pii_type_codes` | VARCHAR(500) | 관련 PII 유형 코드 (DLM PIICODE) |
| `pii_grade` | CHAR(1) | 개인정보 등급 (1/2/3) |
| `affected_rows` | INT | 영향받은 행 수 (Whom 규모) |
| `search_condition` | TEXT | 검색 조건문 (Whom 특정) |
| `sql_text` | TEXT | 실행된 SQL 전문 (선택, 보안 설정에 따라) |
| `access_channel` | VARCHAR(20) | 접근 경로: WEB/WAS/DB_DIRECT/API/BATCH |
| `session_id` | VARCHAR(100) | 세션 ID |
| `result_code` | VARCHAR(10) | 수행 결과: SUCCESS/FAIL/DENIED |
| `hash_value` | VARCHAR(64) | SHA-256 해시 (위변조 방지) |
| `prev_hash` | VARCHAR(64) | 이전 레코드 해시 (해시 체인) |
| `collected_at` | DATETIME | DLM 수집 시간 |
| `partition_key` | VARCHAR(8) | 파티셔닝 키 (YYYYMMDD) |

**인덱스 설계:**
```sql
-- 시간 범위 검색 (가장 빈번한 쿼리)
INDEX idx_access_time (access_time, source_system_id)
-- 사용자별 검색
INDEX idx_user_account (user_account, access_time)
-- 대상 테이블별 검색
INDEX idx_target_table (target_db, target_schema, target_table, access_time)
-- 이상행위 탐지용
INDEX idx_action_type (action_type, access_time)
-- PII 등급별 검색
INDEX idx_pii_grade (pii_grade, access_time)
-- 파티셔닝 (월 단위)
PARTITION BY RANGE (TO_DAYS(access_time))
```

#### M2.2 위변조 방지

| 방법 | 구현 |
|------|------|
| **해시 체인** | 각 레코드의 SHA-256 해시에 이전 레코드 해시를 포함하여 체인 구성 |
| **일일 무결성 검증** | 매일 배치로 해시 체인 무결성 검증, 위변조 시 즉시 알림 |
| **접근 권한 분리** | 접속기록 테이블에 대한 UPDATE/DELETE 권한 제한 (INSERT ONLY) |
| **백업** | 별도 저장소에 주기적 백업 (설정 가능) |

**해시 체인 구조:**
```
Record N: hash = SHA256(log_id + user_account + access_time + action_type + prev_hash)
Record N+1: hash = SHA256(log_id + user_account + access_time + action_type + hash_of_N)
```

#### M2.3 보관기간 관리

| 기능 | 설명 |
|------|------|
| **보관 정책 설정** | 시스템별/등급별 보관기간 설정 (기본: 2년) |

| **파티션 관리** | 월 단위 파티션 자동 생성/삭제 |

#### M2.4 접속기록 검색

| 기능 | 설명 |
|------|------|
| **기간 검색** | 시작일~종료일 범위 검색 |
| **사용자 검색** | 계정/이름/부서별 검색 |
| **대상 검색** | DB/스키마/테이블/컬럼별 검색 |
| **작업유형 검색** | SELECT/UPDATE/DELETE/DOWNLOAD 필터 |

### 3.4 M3. 분석/탐지 (Analysis)

#### M3.1 이상행위 탐지 — 룰 기반

**기본 제공 탐지 규칙:**

| 규칙 ID | 규칙명 | 조건 | 심각도 |
|---------|--------|------|--------|
| R01 | 대량 조회 | 1시간 내 동일 사용자가 N건 이상 조회 (기본: 1000건) | 높음 |
| R02 | 대량 다운로드 | 1일 내 다운로드 건수 N건 초과 (기본: 100건) | 높음 |
| R03 | 업무외 시간 접근 | 22:00~06:00 사이 접근 (설정 가능) | 보통 |
| R04 | 휴일 접근 | 공휴일/주말 접근 | 보통 |
| R05 | 비인가 접근 시도 | result_code = DENIED | 높음 |
| R06 | 고유식별정보 접근 | pii_grade = '1' (1급) 테이블 접근 | 정보 |
| R07 | 반복 조회 | 동일 정보주체를 1시간 내 N회 이상 조회 | 보통 |
| R08 | 민감정보 다운로드 | 1급 민감정보 테이블 다운로드 | 높음 |
| R09 | 신규 IP 접근 | 기존 접속 이력에 없는 IP에서 접근 | 낮음 |
| R10 | 퇴직/이동자 접근 | 비활성 계정으로 접근 시도 | 높음 |

**규칙 설정 UI:**
- 관리자가 규칙 추가/수정/삭제 가능
- 임계치, 시간대, 대상 테이블, 예외 사용자 등 설정
- 규칙별 활성화/비활성화

#### M3.2 이상행위 탐지 — 통계/AI (Phase 3-B)

| 방법 | 설명 |
|------|------|
| **통계 기반** | 사용자별 평균 조회량 대비 표준편차 초과 시 탐지 |

## 9. 경쟁사 대비 차별화 전략

| vs 경쟁사 | DLM Privacy Monitor 차별점 |
|-----------|---------------------------|
| **위즈코리아 (WEEDS)** | 에이전트 없이 DB Audit Log 직접 파싱, 설치/운영 부담 최소화 |
| **이지서티 (UBI SAFER)** | Discovery(PII 탐지) 연계로 모니터링 대상 자동 등록, All-in-One |
| **피앤피시큐어 (INFOSAFER)** | 멀티DB 네이티브 지원 (7개 DBMS), 비식별화+접속기록 통합 |
| **소만사** | DB 레벨 직접 처리, 에이전트리스 경량 구조 |

**핵심 차별점**: 개인정보 탐지(Scanner) → 비식별화(Shield) → 접속기록(Monitor)이 하나의 플랫폼에서 연계되는 **유일한 All-in-One 솔루션**

## 10. 참고: 관련 법령 원문

### 안전성 확보조치 기준 제8조 (접속기록의 보관 및 점검)

> ① 개인정보처리자는 개인정보취급자가 개인정보처리시스템에 접속한 기록을 1년 이상 보관·관리하여야 한다. 다만, 5만명 이상의 정보주체에 관하여 개인정보를 처리하거나, 고유식별정보 또는 민감정보를 처리하는 개인정보처리시스템의 경우에는 2년 이상 보관·관리하여야 한다.
>
> ② 개인정보처리자는 개인정보의 오·남용, 분실·도난·유출·위조·변조 또는 훼손 등에 대응하기 위하여 개인정보처리시스템의 접속기록 등을 월 1회 이상 점검하여야 한다. 특히 개인정보를 다운로드한 것이 발견되었을 경우에는 내부 관리계획으로 정하는 바에 따라 그 사유를 반드시 확인하여야 한다.
>
> ③ 개인정보처리자는 개인정보취급자의 접속기록이 위·변조 및 도난, 분실되지 않도록 해당 접속기록을 안전하게 보관하여야 한다.
