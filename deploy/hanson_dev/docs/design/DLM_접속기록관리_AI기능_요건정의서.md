# DLM 접속기록관리 솔루션 AI 적용 기능 요건 정의서

> 작성일: 2026-04-16
> 목적: 접속기록관리 솔루션에 AI를 적용하기 위한 기능 요건 정리
> 참조: 국내외 경쟁 솔루션 사례, 금융사 도입 현황, 규제 동향

## 1. 시장 현황

### 1.1 국내 경쟁 솔루션 AI 적용 수준

| 벤더 | 솔루션 | AI 적용 수준 |
|------|--------|-------------|
| 피앤피시큐어 | DBSAFER / INFOSAFER | Vision AI 안면인증(ICA), 통계 기반 이상징후 분석 — **국내 AI 선도** |
| 웨어밸리 | Chakra Max / Log Catch | "학습 기반 이상 감지" 표방 — 구체적 ML 기술 미공개 |
| 신시웨이 | PETRA | 규칙 기반 정책 엔진 중심 — AI 미적용 |
| 소만사 | DB-i | 이상징후 탐지 — AI 구체적 적용 미공개 |

**피앤피시큐어 상세:**
- 금융권 DB접근제어 점유율 90% 이상
- ICA(Implicit Continuous Authentication): Vision AI 안면 벡터 분석으로 세션 전 구간 실시간 인증
- INFOSAFER: 사용자 평균 개인정보 사용량 대비 일시적 증가 탐지, 지정 시간 외 조회 탐지
- RT SCANNER: RDBMS/NoSQL에서 주민번호, 신용정보 등 실시간 탐지 (신한은행 도입)
- 다만 접속기록 분석 자체에 ML/딥러닝 적용은 미흡

**웨어밸리 상세:**
- Log Catch: 육하원칙(5W1H) 기반 접속기록 로그 생성/분석
- "학습을 통해 실시간 이상 징후를 자동으로 감지" 표방
- ISMS-P 및 개인정보보호법 인증 요구사항 대응 최적화
- AI/ML 적용에 대한 구체적 기술 상세 미공개

### 1.2 글로벌 벤더 AI 적용 수준

| 벤더 | 솔루션 | 핵심 AI 기능 |
|------|--------|-------------|
| Imperva | Data Risk Analytics (DRA) | ML 기반 행동 분석, 문맥 기반 위험 판단, 자동 우선순위 분류 |
| IBM | Guardium | UBA(사용자 행동 분석), AI 데이터 분류, DDR(Data Detection & Response) |
| Varonis | Next-Gen DAM (2025 출시) | UEBA 기반 자동 탐지/차단, 자동 권한 회수, 에이전트리스 아키텍처 |
| Securonix | UEBA Platform | ML 기반 UEBA, False Positive 대폭 감소, 금융 특화 유즈케이스 |

**Imperva DRA 상세:**
- 사용자 데이터 접근 패턴을 지속 학습하여 베이스라인 구축
- 접근 데이터 민감도 + 사용자 역할 + 접근 방식을 종합 분석하여 단순 이상치와 실제 위협 구분
- 수억 개 데이터 접근 이벤트에서 의심 활동 자동 식별/분류

**Varonis Next-Gen DAM 상세:**
- 에이전트리스 아키텍처로 수천 개 DB 모니터링
- UEBA로 의심 활동 자동 탐지/차단
- 과도한 권한 자동 회수, 민감 데이터 마스킹
- Databricks, SQL Server, RDS, Postgres, Oracle, Snowflake 등 지원

### 1.3 핵심 결론

> **국내 접속기록관리 시장에서 ML/AI 기반 행동 분석을 본격 적용한 솔루션은 아직 없다.**
> Imperva DRA, Varonis Next-Gen DAM 수준의 ML 기반 UEBA는 국내에 부재하며, 이것이 DLM의 차별화 기회이다.

## 2. 금융권 규제 동향

### 2.1 개인정보보호법 - 안전성 확보조치 기준 (2024.10)

- 접속기록 보관: 일반 1년 이상, 5만명 이상 또는 고유식별/민감정보 처리 시 2년 이상
- 접속기록 점검: 개인정보 보호책임자가 연 1회 이상 접근 권한 관리, 접속기록 보관 및 점검 실태를 점검
- **이상행위 점검 기준**: 과도한 접속 성공/실패, 부적절한 명령어 등 패킷 포함
- 공공시스템 운영기관은 접속기록의 저장/분석/점검/관리 등 추가 조치 의무

### 2.2 전자금융감독규정 (2025.2 전면 개정)

- 정보처리시스템 가동기록: 접속 성공 여부와 무관하게 자동 기록/유지, 1년 이상 보존
- 이용자정보 조회 기록 1년 이상 보존
- **"규칙(Rule) → 원칙(Principle) 중심" 자율보안 체제로 전환**
- AI 활용 확대를 위한 규제 샌드박스 운영

### 2.3 금융분야 AI 가이드라인 (2026 Q1 시행 예정)

- 7대 핵심 AI 원칙: 거버넌스, 적법성, 보충성, 신뢰성, 금융 안정성, 선의, 보안
- 금융권 AI 플랫폼 구축 (2025년 상반기)
- 금융권 특화 한글 말뭉치 단계적 제공

### 2.4 시사점

현행 규정은 "이상행위 점검"을 의무화하고 있으나, AI/ML 기술 사용을 명시적으로 요구하지는 않는다.
다만 점검 대상 데이터가 대규모화됨에 따라 AI/ML 없이는 실효성 있는 점검이 어려워지고 있어, **사실상 AI 도입이 불가피한 상황**이다.

## 3. DLM 현재 AI 구현 상태

| 기능 | 상태 | AI 여부 | 구현 위치 |
|------|------|---------|----------|
| PII 자동탐지 | ✅ 운영 | ✅ LLM 기반 | DiscoveryEngineImpl.java + Privacy-AI |
| 가중치 스코어링 | ✅ 운영 | ✅ Meta 40% + Pattern 35% + AI 25% | DiscoveryEngineImpl.java |
| 이상행위 탐지 | ✅ 운영 | ❌ 규칙 기반 7가지 유형 | AccessLogDetectionEngineImpl.java |
| 접속기록 수집 | ✅ 운영 | ❌ 데이터 수집만 | AccessLog 관련 |
| PSM Agent SQL 캡처 | ✅ 운영 | ❌ 수집만 | dlm-agent |
| 접속기록 보고서 | ❌ 미구현 | - | - |
| 로그 분석 AI | ❌ 미구현 | - | - |
| UEBA | ❌ 미구현 | - | - |
| 권한 최적화 | ❌ 미구현 | - | - |

**현재 이상행위 탐지 7가지 규칙** (AccessLogDetectionEngineImpl.java):
1. VOLUME — 시간 윈도우 내 대량 접속 감지
2. TIME_RANGE — 야간/공휴일 시간대 접속 탐지
3. ACCESS_DENIED — 접속 거부 반복 패턴
4. PII_GRADE — 고등급 PII 대량 접근 감지
5. REPEAT — 동일 테이블 반복 접근
6. NEW_IP — 미등록 IP 접근 (90일 이력 기반)
7. INACTIVE — 미사용 계정 감지

## 4. AI 적용 기능 요건 (8개 영역)

### F1. ML 기반 이상행위 탐지 (Anomaly Detection)

**현재**: 규칙 기반 임계값 방식 → **목표**: ML 기반 베이스라인 학습 + 이상 스코어링

| 항목 | 내용 |
|------|------|
| 개요 | 사용자별 접속 패턴을 학습하여 베이스라인을 생성하고, 이탈 시 자동 탐지 |
| 기술 | Isolation Forest, Autoencoder, 시계열 분석 (접속 빈도/시간/데이터량) |
| 입력 데이터 | AccessLog (시간, 사용자, IP, 쿼리 유형, 조회 건수) |
| 출력 | 이상 스코어(0-100), 이상 사유, 관련 로그 ID |
| 기대 효과 | 규칙으로 잡지 못하는 진화형/복합적 이상행위 탐지 |
| 글로벌 벤치마크 | Imperva DRA, IBM Guardium UBA |

**세부 탐지 시나리오:**
- 평소 대비 10배 이상 데이터 조회량 급증
- 접속 시간대 패턴 이탈 (항상 9-18시 접속하던 사용자가 새벽 접속)
- 접속 테이블 패턴 이탈 (처음 접근하는 민감 테이블)
- 쿼리 유형 패턴 이탈 (항상 SELECT만 하던 사용자가 DDL 실행)
- 복합 이상: 개별적으로는 정상이지만 조합하면 의심스러운 패턴

**구현 방향:**
- 기존 규칙 기반 탐지(7가지)는 유지하고, ML 기반 탐지를 **추가 레이어**로 적용
- Privacy-AI(FastAPI)에 ML 모델 서빙 엔드포인트 추가
- 학습 주기: 일 1회 베이스라인 갱신 (최근 30~90일 데이터 기반)

---

### F2. 사용자 행동 분석 (UEBA)

**현재**: 미구현 → **목표**: 사용자/엔티티 프로파일링 + 피어그룹 비교

| 항목 | 내용 |
|------|------|
| 개요 | 사용자별 행동 프로파일을 생성하고, 동일 직무 그룹 대비 이탈자 탐지 |
| 기술 | 클러스터링 (K-Means), 피어그룹 분석, 리스크 스코어 누적 |
| 입력 데이터 | AccessLog + 사용자 정보(부서/직급/역할) |
| 출력 | 사용자별 리스크 스코어, 피어그룹 이탈도, 행동 변화 트렌드 |
| 기대 효과 | 내부자 위협 조기 탐지, 퇴직 예정자 이상 접근 감시 |
| 글로벌 벤치마크 | Securonix UEBA, Exabeam |

**세부 기능:**
- 사용자 프로파일 카드 (평균 접속 빈도, 주 접근 테이블, 시간대 분포)
- 피어그룹 대비 이탈도 시각화 (같은 부서/역할 사용자 대비)
- 리스크 스코어 추이 대시보드 (누적 점수 시계열)
- 고위험 사용자 Top-N 리스트
- 행동 변화 알림 (최근 7일 vs 이전 30일 패턴 비교)

**탐지 가능 이상행위:**
- 비정상적 로그인 시간 (야간/휴일)
- 평소와 다른 IP/위치에서의 접속
- 단시간 내 다수 인증 실패
- 비정상적 대량 데이터 전송/조회
- 권한 범위 외 테이블/DB 접근 시도
- 퇴직 예정자의 비정상적 데이터 접근 증가

### F3. SQL 위험도 자동 분류

**현재**: PSM Agent로 SQL 캡처만 → **목표**: AI 기반 자동 위험도 분류

| 항목 | 내용 |
|------|------|
| 개요 | 캡처된 SQL을 자동 분석하여 위험도를 분류하고 스코어링 |
| 기술 | 규칙 기반 분류 + LLM 기반 의도 분석 (하이브리드) |
| 입력 데이터 | PSM Agent 캡처 SQL, 대상 테이블의 PII 등급 |
| 출력 | SQL 위험도 (Critical/High/Medium/Low), 위험 사유 |

**위험도 분류 체계:**

| 등급 | 유형 | 예시 |
|------|------|------|
| Critical | DDL 변경 | DROP TABLE, TRUNCATE, ALTER TABLE |
| High | 대량 추출 | SELECT * (WHERE 없음), 민감 테이블 FULL SCAN, UNION 기반 공격 |
| Medium | 민감 데이터 접근 | PII 테이블 조회, 개인정보 컬럼 SELECT |
| Low | 일반 업무 | 정상적 CRUD 쿼리 |

**구현 방향:**
- 1차: 규칙 기반 SQL 파싱으로 DDL/DML 유형 분류 + WHERE 절 유무 + 대상 테이블 PII 등급 조합
- 2차: LLM으로 SQL 의도 분석 (대량 데이터 유출 시도 여부, SQL injection 패턴 등)
- PSM Agent 로그와 AccessLog를 조인하여 "누가, 어떤 SQL을, 어떤 결과로" 실행했는지 통합 분석

### F4. 자연어 기반 접속기록 분석 (LLM)

**현재**: Privacy-AI LLM 인프라 존재 → **목표**: 접속기록 분석용 LLM 기능 추가

| 항목 | 내용 |
|------|------|
| 개요 | LLM을 활용하여 접속기록을 자연어로 질의/분석/요약 |
| 기술 | 기존 Privacy-AI LLM 엔진 확장 (새 엔드포인트 추가) |
| 입력 | 자연어 질의 + 관련 접속기록 데이터 |
| 출력 | 자연어 분석 결과, 요약 보고서, 이상행위 설명 |

**활용 시나리오:**
- "지난 24시간 동안 비정상적인 접속 패턴이 있었나요?"
- "이 사용자의 최근 1주일 접속 패턴을 분석해줘"
- 이상행위 Alert에 대한 자연어 설명 자동 생성
  - 예: "이 사용자는 평소 대비 15배 많은 고객 테이블 조회를 수행했으며, 이는 지난 90일 패턴과 크게 이탈합니다"
- 월간/분기별 접속기록 분석 보고서 자동 생성
- 감사 시 자연어 기반 로그 검색

**구현 방향:**
- Privacy-AI에 `/api/v1/accesslog/analyze` 엔드포인트 추가
- DLM이 질의 관련 접속기록 데이터를 JSON으로 추출하여 LLM에 context로 전달
- LLM이 데이터를 분석하고 자연어 응답 반환

### F5. 개인정보 탐지 고도화

**현재**: LLM PII 탐지 운영 중 → **목표**: 증분 스캔 + 접속기록 연계

| 항목 | 내용 |
|------|------|
| 증분 스캔 | 신규/변경 테이블만 재탐지 (DDL 변경 감지 기반) |
| 접속기록 연계 | PII 탐지 결과를 접속기록 이상행위 탐지 가중치에 자동 반영 |
| 자동 마스킹 추천 | 탐지된 PII 테이블에 대한 마스킹 정책 자동 추천 |
| 탐지 정확도 피드백 | 관리자 확인/수정 결과를 학습 데이터로 활용하여 정확도 향상 |

---

### F6. 접근권한 최적화 (Least Privilege)

**현재**: 미구현 → **목표**: 미사용 권한 식별 + 최소권한 추천

| 항목 | 내용 |
|------|------|
| 개요 | 접속기록 분석으로 실제 사용 권한과 부여 권한 간 갭을 식별 |
| 기술 | 접속기록 통계 분석 + DB 권한 메타데이터 매핑 |
| 출력 | 미사용 권한 리포트, 최소권한 추천, 권한 회수 대상 리스트 |

**세부 기능:**
- "이 사용자는 90일간 해당 테이블에 접근하지 않았습니다" 알림
- 부서/역할별 필요 최소 권한 자동 산출
- 과잉 권한 현황 대시보드
- 권한 변경 이력 추적

---

### F7. 컴플라이언스 자동 점검

**현재**: 미구현 → **목표**: 규정별 준수 여부 자동 점검

| 항목 | 내용 |
|------|------|
| 개요 | 개인정보보호법, 전자금융감독규정 등 요구사항을 자동 점검 |
| 점검 항목 | 접속기록 보관 기간, 이상행위 점검 이력, 접근권한 관리 현황 |
| 출력 | 규정별 준수/미준수 현황, 미준수 항목 자동 알림, 감사 보고서 |

**규정 매핑:**

| 규정 | 요구사항 | 자동 점검 항목 |
|------|----------|---------------|
| 개인정보보호법 | 접속기록 1~2년 보관 | 보관 기간 자동 관리 + 만료 알림 |
| 전자금융감독규정 | 가동기록 1년 보관, 이용자정보 조회 기록 | 보관 현황 자동 모니터링 |
| 신용정보법 | 접근권한 관리, 접속기록 보관 | 미사용 권한 자동 식별 |
| ISMS-P | 접근통제, 접속기록 관리 | 정책 준수 현황 자동 점검 |

> **ISMS-P (정보보호 및 개인정보보호 관리체계 인증)**
> 한국인터넷진흥원(KISA)이 운영하는 국내 정보보호 인증 제도로, 기존 ISMS(정보보호 관리체계)와 PIMS(개인정보보호 관리체계)를 2018년 통합하여 시행 중이다. 일정 규모 이상의 기업(ISP, 매출 100억 이상, 이용자 100만 이상 등)은 **법적 의무 인증** 대상이며, 접속기록 관리는 인증 심사 시 필수 점검 항목에 해당한다.

### F8. 접속기록 보고서 (Compliance & Audit Reporting)

**현재**: 미구현 → **목표**: 법적 의무 보고서 자동 생성 + AI 기반 분석 보고서

#### 법적 근거

| 규정 | 보고서 관련 의무 |
|------|-----------------|
| **개인정보보호법 안전성확보조치 기준 제8조** | 접속기록 **월 1회 이상 점검** 의무, 이상행위 점검 및 조치 결과 기록 |
| **개인정보보호법 시행령** | 개인정보 보호책임자가 **연 1회 이상** 접근권한 관리·접속기록 보관·점검 실태를 점검·관리 |
| **전자금융감독규정 (2025.2 개정)** | 정보처리시스템 가동기록 보관 현황 보고, IT감사 결과보고 의무 |
| **금감원 IT감사 가이드라인** | IT감사 6단계 중 **⑤결과보고** 필수 — IT실태평가 시 이행여부 기준 활용 예정 |
| **ISMS-P** | 접근통제·접속기록 관리 현황 — 보안 인증 감사 자료 |

#### 경쟁 솔루션 보고서 현황

| 벤더 | 보고서 기능 |
|------|-----------|
| **피앤피시큐어 DBSAFER** | **약 93종** 보고서 템플릿, PDF/HTML/HWP/DOC/XML/XLS/TXT 변환 지원 |
| **피앤피시큐어 INFOSAFER** | 공공/금융/이커머스 산업별 최적화 점검보고서, 각종 평가·인증·감사 증빙 활용 |
| **Imperva DAM** | SOX/PCI/HIPAA/GDPR/NYDFS 등 규정별 사전 구성 보고서 + 커스텀 보고서 빌더 |
| **Varonis** | **50개 내장 규칙 + 400개 패턴** 기반 HIPAA/SOX/PCI/GDPR 컴플라이언스 보고서 |

> **결론**: 보고서는 선택이 아닌 **법적 의무**이며, 경쟁 솔루션은 모두 핵심 기능으로 제공 중이다. DLM에 보고서 기능이 없으면 금융사 도입 시 결격 사유가 된다.

**R1. 접속기록 정기 점검 보고서** (법적 의무 — 월 1회)

| 항목 | 내용 |
|------|------|
| 법적 근거 | 개인정보보호법 안전성확보조치 기준 제8조 |

- 개인정보취급자 수 및 접속 횟수 현황
- 업무시간 외 접속 현황 (야간/주말/공휴일)
- 동일 정보주체 대량 중복 조회 현황 (10회 이상)

- 접근권한 오·남용 및 정보 유출 의심 건
- 개인정보 다운로드 기록 및 사유 확인 결과
- 이상행위 탐지 건수 및 조치 결과

| 항목 | 내용 |

## 6. 금융사 도입 현황

### 국내 금융권

- 대부분 DBSAFER(피앤피시큐어) 기반 DB접근제어 운영
- 이상징후 분석은 주로 규칙 기반(Rule-based) 임계값 방식
- AI/ML 기반 고도화된 이상행위 탐지는 아직 보편화되지 않음
- FDS(이상거래탐지)에는 AI 활발하나, 내부 DB접근기록 분석에는 제한적
- 신한은행: DBSAFER DB + RT SCANNER 도입 (실시간 민감정보 탐지)

### 글로벌 금융권 트렌드

- 54% 이상의 조직이 내부자 위협 탐지에 AI/ML 사용 중 또는 계획
- 내부자 위험 예산: IT 보안 예산의 16.5% (2023년 8.2% 대비 2배 증가)
- 탐지 시간: 전통적 방식 평균 77일 → UEBA 도입 시 실시간 탐지
- Gartner: "DAM 에이전트를 교체하고 DB에서 앱, AI 에이전트까지 보안 범위를 확장할 때"

## 7. 핵심 시사점

1. **국내 시장 빈 공간**: 접속기록관리에 본격적인 ML/AI를 적용한 국내 솔루션은 없다. 피앤피시큐어가 안면인증(ICA)으로 AI를 선도하지만, 접속기록 분석 자체의 AI화는 미흡하다.

2. **글로벌 트렌드**: Gartner는 DAM → DSP(Data Security Platform) 전환을 권고하며, ML 기반 UEBA가 핵심 요소로 자리잡았다. Imperva, Varonis, IBM이 이 방향을 선도한다.

3. **금융 규제 방향**: 2025년 전자금융감독규정 전면 개정으로 "규칙 → 원칙 중심" 자율보안 체제로 전환 중이며, AI 활용 확대를 위한 규제 샌드박스가 운영되고 있다.

4. **DLM의 강점**: 이미 LLM 기반 PII 탐지 + PSM Agent SQL 캡처 + 규칙 기반 이상행위 탐지 인프라가 갖춰져 있어, ML 기반 이상행위 탐지와 LLM 기반 로그 분석 추가가 자연스러운 다음 단계이다.
