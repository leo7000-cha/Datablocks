## SCENE 1. 대시보드 (Dashboard)

### 화면
메뉴: **접속기록관리 > 대시보드**

### 시연 포인트
1. **상단 통계 카드** 확인
   - 오늘 접속 건수 / 전체 접속 건수 (56건)
   - 미처리 알림 수 (실시간 배지)
   - 수집 소스 상태 (3개 등록, 1개 활성)
   - 해시 검증 상태 (VALID)

2. **컴플라이언스 스코어카드** (법규 준수 현황)
   - 개인정보보호법 제8조 2항: 접속기록 2년 보관 → **적합**
   - 제8조 3항: 월 1회 점검 → **적합** (이번 달 3회)
   - 제8조 4항: 위변조 방지 → **적합** (SHA-256 해시 체인)
   - 제5조: 비인가 접근 탐지 → **적합** (8개 규칙 활성)
   - 전자금융감독규정 제14조: 1년 이상 보존 → **적합** (2년 설정)

3. **차트** - 시간대별 접속 추이, 작업 유형 분포

### 화법
> "대시보드에서 법규 준수 현황을 한눈에 확인할 수 있습니다.
> 개인정보보호법과 전자금융감독규정의 주요 항목이 모두 '적합' 상태입니다."

## SCENE 2. 접속기록 조회

### 화면
메뉴: **접속기록관리 > 접속기록**

### 시연 포인트
1. **날짜 범위 선택** (필수) → 2026-04-16 ~ 2026-04-17
2. **결과 확인** - 55건 표시
   - 접속일시, 사용자(COTDL), IP, 작업유형(SELECT), 대상DB(DAON), 대상테이블(ACTEUR), PII등급(1등급)
3. **필터 테스트**
   - 사용자 필터: "COTDL" 입력 → 55건
   - PII등급 필터: "1등급" 선택 → 55건 (모두 고등급 PII)
   - 작업유형 필터: "SELECT" → 55건
4. **상세 보기** - 로그 클릭 → 상세 정보 (SQL문, 해시값, 이전해시 등)
5. **페이지네이션** - 페이지 이동 확인

### 화법
> "접속기록은 반드시 날짜 범위를 지정해야 조회됩니다.
> 개인정보 접근 등급이 1~3등급으로 자동 분류되어, 고등급 개인정보 접근을 바로 식별할 수 있습니다."

## SCENE 4. 소명 워크플로우 (핵심 시연)

### 화면
알림 목록 → 소명 요청 → 소명 제출 페이지 → 관리자 승인/반려

### 시연 포인트 (전체 플로우 라이브)

**Step 1: 관리자 → 소명 요청 발송**
1. 알림 목록에서 "NEW" 또는 "ESCALATED" 상태 알림 선택
2. "소명 요청" 버튼 클릭
3. 대상자 이메일 입력 → 발송
4. 상태가 "NOTIFIED"로 변경
5. 이메일 발송 (72시간 유효 토큰 포함)

**Step 2: 대상자 → 소명 제출 (로그인 불필요)**
1. 이메일 링크 클릭 → 소명 제출 페이지
2. 소명 사유 입력: "업무상 필요한 접근이었습니다. 개인정보 조회 업무를 위한 정상 접근입니다."
3. 제출 → 상태 "JUSTIFIED"로 변경

**Step 3-A: 관리자 → 승인**
1. 관리자가 소명 내용 확인
2. "승인" → 상태 "RESOLVED" (완결)
3. 승인 코멘트: "소명 내용 확인, 정상 업무로 판단됨"

**Step 3-B: 관리자 → 반려 (재소명 요청)**
1. "반려" → 상태 "RE_JUSTIFY"
2. 반려 사유: "소명 내용이 불충분합니다. 구체적인 사유를 재제출해주세요."
3. 새 이메일 자동 발송 → 대상자 재제출

**SLA 관리 (자동)**
- 48시간 내 미응답 → OVERDUE 자동 전환
- 72시간 초과 → ESCALATED 자동 에스컬레이션
- 관리자에게 경고 이메일 발송

### 화법
> "개인정보보호법 안전성확보조치 제29조에 따라, 이상행위 발생 시 대상자에게 소명을 요청하고
> 관리자가 승인/반려 할 수 있는 전체 워크플로우를 제공합니다.
> SLA(응답 기한)가 설정되어 있어 미응답 시 자동으로 에스컬레이션됩니다."

## SCENE 6. 예외 규칙 관리 (Suppression)

### 화면
메뉴: **접속기록관리 > 예외 규칙**

### 시연 포인트
1. **예외 규칙 목록** 확인 (현재 5건 활성)
2. **예외 등록** - "R01 대량접속 + admin 사용자" → 유효기간 2026-05-31, 검토주기 30일
3. **일괄 무시 + 자동 예외 등록** (dismiss-with-suppression)
   - 특정 규칙+사용자의 모든 알림을 무시하면서 동시에 예외 규칙 자동 생성
4. **정기 검토** - 30일마다 예외 규칙 유효성 검토
5. **비활성화** - 필요 없어진 예외 해제
6. **감사 추적** - 예외 규칙의 생성/연장/검토/비활성화 이력

### 화법
> "안전성확보조치 기준에 따라, 오탐으로 판단된 알림에 대해 예외 규칙을 등록할 수 있습니다.
> 예외 규칙은 유효기간이 있고, 정기 검토가 의무화되어 관리자의 방치를 방지합니다."

### 시연 포인트
1. **등록된 소스 3개** 확인
   - DB_AUDIT (Oracle 감사로그 수집) - RUNNING
   - WAS_AGENT (Java Agent 연동) - 대기 중
   - DB_DAC (접근통제 솔루션 연동) - 대기 중

2. **소스 등록** 설명

4. **수집 시작/중지** - 일괄 시작/중지 가능

5. **Agent 연동** 설명
   - PSM(Java Agent)가 WAS에서 JDBC 쿼리를 가로채서 DLM으로 전송
   - 실시간 접속기록 수집

### 화법
> "3가지 방식으로 접속기록을 수집합니다: DB 감사로그, Java Agent, 접근통제 솔루션.

## SCENE 10. 감사 정책 설정

### 화면
메뉴: **접속기록관리 > 감사 정책**

### 시연 포인트
1. **DB 선택** → DAON (Oracle)
2. **PII 테이블 목록** 자동 로드 (메타데이터 기반)
3. **대상 테이블 선택** (체크박스)
4. **감사 스크립트 자동 생성** 클릭

```sql
-- DLM 감사 정책 스크립트 (자동 생성)
CREATE AUDIT POLICY XAUDIT_PII_POLICY
  ACTIONS
    SELECT ON COOWNSER.ACTEUR,
    INSERT ON COOWNSER.ACTEUR,
    UPDATE ON COOWNSER.ACTEUR,
    DELETE ON COOWNSER.ACTEUR;
AUDIT POLICY XAUDIT_PII_POLICY;
```

5. **DBA에게 전달** → Oracle DB에 적용

### 화법
> "DLM의 PII 메타데이터를 활용하여, 개인정보가 포함된 테이블에 대한
> Oracle 감사 정책 스크립트를 자동으로 생성합니다.
> DBA가 이 스크립트를 실행하면 자동으로 접속기록이 수집됩니다."

## 시연 시 Q&A 예상 질문

**Q1: 접속기록 보관 기간은?**
> 기본 2년 설정. 금융권 요구사항(5년)에 맞게 조정 가능. 파티션 기반 자동 아카이브.

**Q2: 해시 체인이 깨지면?**
> INVALID 표시되며 최초 변조 레코드 ID를 알려줍니다. 관리자에게 자동 알림.

**Q3: 대량 데이터 성능은?**
> 월 단위 파티셔닝. 3개월 미래 파티션 자동 생성. 보관 기간 초과 시 자동 삭제.

**Q4: 기존 접근통제 솔루션과 연동?**
> DB_DAC 소스 타입으로 커스텀 SQL 쿼리를 지정하여 기존 솔루션의 로그를 수집 가능.

**Q5: 에이전트 설치가 필요한가?**
> WAS 에이전트는 Java Agent(BCI) 방식으로 JVM 옵션 추가만으로 설치. 코드 수정 불필요.
