# DLM 패치 배포 가이드

> 초기 배포 완료 후, 코드 수정/버그 수정/기능 추가 시 패치를 적용하는 절차
> 작성일: 2026-04-08

---

## 1. 개요

### 초기 배포 vs 패치 배포

| 구분 | 초기 배포 | 패치 배포 |
|------|-----------|-----------|
| DB 초기화 | 필요 (SQL 실행) | 불필요 (기존 DB 유지) |
| Docker 설치 | 필요 (폐쇄망 RPM) | 불필요 (이미 설치됨) |
| 환경변수 설정 | 필요 (.env 수정) | 불필요 (기존 설정 유지) |
| 이미지 교체 | 최초 로드 | **교체 (load → restart)** |
| 다운타임 | 최초 구동 | **약 30초~1분** |

> 패치 배포는 **이미지만 교체**하면 되므로, 초기 배포보다 훨씬 간단합니다.

## 2. 패치 유형 판단

패치 작업 전, 어떤 유형의 변경인지 먼저 확인하세요:

| 유형 | 설명 | 필요한 파일 | 개발PC 빌드 | 서버 작업 |
|------|------|-------------|-------------|-----------|
| **A. DLM 코드만** | Java 코드, JSP, 설정 변경 | `dlm-app.tar.gz` | DLM 이미지 빌드 | 이미지 교체 |
| **B. Privacy-AI만** | Python 코드 변경 | `dlm-privacy-ai.tar.gz` | AI 이미지 빌드 | 이미지 교체 |
| **C. 둘 다** | DLM + Privacy-AI 모두 변경 | 두 파일 모두 | 둘 다 빌드 | 두 이미지 교체 |
| **D. DB 스키마 변경** | 테이블/컬럼 추가·수정 | 이미지 + SQL 패치 파일 | 빌드 + SQL 작성 | 이미지 교체 + SQL 실행 |
| **E. 설정만** | .env 값 변경 (포트, DB 주소 등) | 없음 | 불필요 | .env 수정 후 재시작 |

## 3. 개발PC 작업 (WSL 환경)

### STEP 1: 코드 수정 및 테스트

```bash
# 개발환경에서 코드 수정 후 로컬 테스트
cd /app/Datablocks
docker compose up -d --build    # 개발용 전체 빌드
```

정상 동작 확인 후 다음 단계로 진행합니다.

### STEP 2: 이미지 빌드

```bash
cd /app/Datablocks

# --- 유형 A: DLM만 빌드 ---
docker compose build dlm

# --- 유형 B: Privacy-AI만 빌드 ---
docker compose build dlm-privacy-ai

# --- 유형 C: 둘 다 빌드 ---
docker compose build dlm dlm-privacy-ai
```

### STEP 3: 이미지 추출 (tar.gz)

```bash
cd /app/Datablocks

# --- DLM 이미지 추출 ---
docker save datablocks-dlm:latest | gzip > dlm-app.tar.gz

# --- Privacy-AI 이미지 추출 ---
docker save datablocks-dlm-privacy-ai:latest | gzip > dlm-privacy-ai.tar.gz
```

> 파일 크기 참고: DLM ~301MB, Privacy-AI ~78MB

### STEP 4: 빌드 확인

```bash
# 파일이 정상 생성되었는지 확인
ls -lh dlm-app.tar.gz dlm-privacy-ai.tar.gz
```

출력 예시:
```
-rw-r--r-- 1 user user 301M Apr  8 10:30 dlm-app.tar.gz
-rw-r--r-- 1 user user  78M Apr  8 10:31 dlm-privacy-ai.tar.gz
```

### STEP 5: 고객사 배포 폴더에 복사

```bash
# 대상 고객사 폴더의 images/ 디렉토리에 복사
# 예시: 한국손사
cp dlm-app.tar.gz deploy/hanson/images/
cp dlm-privacy-ai.tar.gz deploy/hanson/images/    # Privacy-AI도 변경된 경우

# 예시: iM캐피탈
cp dlm-app.tar.gz deploy/imcapital/images/

# 예시: JB우리캐피탈
cp dlm-app.tar.gz deploy/jbwoori/images/
```

### STEP 6: DB 패치 SQL 준비 (유형 D인 경우만)

DB 스키마가 변경된 경우, 패치 SQL 파일을 작성합니다:

```bash
# 패치 SQL 파일 작성 (파일명 규칙: PATCH_YYYYMMDD_설명.sql)
vi deploy/hanson/mariadb/PATCH_20260408_ADD_COLUMN.sql
```

패치 SQL 예시:
```sql
-- PATCH_20260408_ADD_COLUMN.sql
-- 설명: OOO 테이블에 XXX 컬럼 추가
-- 주의: 이 파일은 한 번만 실행하세요

ALTER TABLE "SOME_TABLE" ADD COLUMN "NEW_COLUMN" VARCHAR(100) DEFAULT NULL;
```

### STEP 7: Windows로 복사 → USB

WSL에서 Windows 탐색기로 파일을 복사합니다:

```bash
# WSL → Windows 바탕화면으로 복사 (예시)
cp -r deploy/hanson/ /mnt/c/Users/사용자명/Desktop/hanson-patch/
```

또는 Windows 탐색기에서 `\\wsl$\Ubuntu\app\Datablocks\deploy\hanson\` 경로로 직접 접근하여 복사합니다.

**USB에 담을 파일 목록:**

```
USB/
├── images/
│   ├── dlm-app.tar.gz              ← 항상 포함
│   └── dlm-privacy-ai.tar.gz      ← Privacy-AI 변경 시만
└── mariadb/                         ← DB 패치 시만
    └── PATCH_20260408_ADD_COLUMN.sql
```

> 전체 배포 폴더가 아닌 **변경된 파일만** USB에 담으면 됩니다.

## 4. 고객사 서버 작업

### STEP 1: 파일 업로드

USB의 파일을 서버로 복사합니다. 작업 디렉토리 예시: `/tmp/patch/`

```bash
# USB 마운트 (서버 환경에 따라 다름)
mkdir -p /tmp/patch
cp -r /media/usb/* /tmp/patch/

# 파일 확인
ls -lh /tmp/patch/images/
```

### STEP 2: 컨테이너 중지

```bash
cd /app/Datablocks

# 현재 상태 먼저 확인
docker ps

# 컨테이너 중지
docker compose -f docker-compose.고객사.yml down
```

> **고객사별 compose 파일명:**
> - 한국손사: `docker-compose.hanson.yml`
> - iM캐피탈: `docker-compose.imcapital.yml`
> - JB우리캐피탈: `docker-compose.jbwoori.yml`

### STEP 3: 이미지 로드

```bash
# DLM 이미지 로드
docker load < /tmp/patch/images/dlm-app.tar.gz

# Privacy-AI 이미지도 변경된 경우
docker load < /tmp/patch/images/dlm-privacy-ai.tar.gz
```

출력 확인:
```
Loaded image: datablocks-dlm:latest               ← 이 메시지가 나오면 OK
Loaded image: datablocks-dlm-privacy-ai:latest     ← 이 메시지가 나오면 OK
```

### STEP 4: 컨테이너 재시작

```bash
cd /app/Datablocks
docker compose -f docker-compose.고객사.yml up -d
```

### STEP 5: 동작 확인

```bash
# 1. 컨테이너 상태 확인 (둘 다 Up 상태여야 함)
docker ps

# 2. DLM 로그 확인 (에러 없이 Started 메시지 나오는지)
docker logs -f dlm-app --tail 30

# 3. 브라우저 접속 확인
#    http://서버IP:8080
```

정상 출력 예시:
```
dlm-app          Up 1 minute    0.0.0.0:8080->8080/tcp
dlm-privacy-ai   Up 1 minute    0.0.0.0:8000->8000/tcp
```

## 5. DB 패치가 있는 경우 (유형 D)

이미지 교체 **후**, DB 패치 SQL을 실행합니다.

### 외부 MariaDB인 경우 (한국손사, iM캐피탈, JB우리캐피탈)

```bash
# DB 서버에서 실행
mysql -u cotdl -p cotdl < /tmp/patch/mariadb/PATCH_20260408_ADD_COLUMN.sql
```

### Docker MariaDB인 경우 (개발환경)

```bash
docker exec -i dlm-mariadb mysql -u cotdl -p'비밀번호' cotdl < /tmp/patch/mariadb/PATCH_20260408_ADD_COLUMN.sql
```

### 실행 후 확인

```bash
# 패치 내용에 따라 확인 (예: 컬럼 추가)
mysql -u cotdl -p cotdl -e "DESC SOME_TABLE;"
```

## 6. 설정만 변경하는 경우 (유형 E)

이미지 교체 없이 환경변수만 수정하는 경우:

```bash
cd /app/Datablocks

# 1. 환경변수 파일 수정
vi .env.고객사

# 2. 컨테이너 재시작
docker compose -f docker-compose.고객사.yml down
docker compose -f docker-compose.고객사.yml up -d
```

## 7. 확인 체크리스트

패치 적용 후 아래 항목을 확인하세요:

```
개발PC
  ☐ 코드 수정 완료 + 로컬 테스트 통과
  ☐ docker compose build 성공
  ☐ docker save + gzip 완료
  ☐ tar.gz 파일 크기 정상 (DLM ~301MB, AI ~78MB)
  ☐ USB에 파일 복사 완료

고객사 서버
  ☐ docker compose down 실행
  ☐ docker load 성공 ("Loaded image: ..." 메시지 확인)
  ☐ docker compose up -d 실행
  ☐ docker ps → 컨테이너 2개 모두 Up 상태
  ☐ docker logs dlm-app → 에러 없음
  ☐ 브라우저 http://서버IP:8080 접속 → 로그인 성공
  ☐ 변경된 기능 동작 확인

DB 패치 (해당 시)
  ☐ SQL 파일 실행 완료
  ☐ 스키마 변경 확인 (DESC, SHOW TABLES 등)
  ☐ 애플리케이션에서 관련 기능 정상 동작
```

## 8. 롤백 (문제 발생 시)

패치 후 문제가 발생하면 이전 이미지로 되돌릴 수 있습니다.

### 사전 준비 (패치 전에 미리 백업)

```bash
# 패치 적용 전, 현재 이미지를 백업해 두세요
cd /app/Datablocks
docker save datablocks-dlm:latest | gzip > dlm-app-backup-$(date +%Y%m%d).tar.gz
```

### 롤백 수행

```bash
cd /app/Datablocks

# 1. 현재 컨테이너 중지
docker compose -f docker-compose.고객사.yml down

# 2. 백업 이미지 복원
docker load < dlm-app-backup-20260408.tar.gz

# 3. 재시작
docker compose -f docker-compose.고객사.yml up -d
```

> **주의**: DB 스키마 변경(유형 D)이 포함된 패치는 롤백이 복잡합니다.
> DB 패치 전에 반드시 DB 백업을 수행하세요:
> ```bash
> mysqldump -u cotdl -p cotdl > cotdl-backup-$(date +%Y%m%d).sql
> ```

## 9. 자주 묻는 질문

### Q: 패치 적용 시 DB 데이터가 날아가나요?
**아니요.** DB는 Docker 외부(또는 별도 볼륨)에 있으므로 이미지 교체와 무관합니다.

### Q: .env 파일도 다시 설정해야 하나요?
**아니요.** 이미지 교체만 하면 기존 .env 설정이 그대로 유지됩니다.

### Q: 서비스 중단 시간은 얼마나 되나요?
`down` → `load` → `up` 과정에서 약 **30초~1분** 정도 중단됩니다.

### Q: DLM만 패치하고 Privacy-AI는 그대로 두어도 되나요?
**네.** 변경된 이미지만 교체하면 됩니다. 변경되지 않은 이미지는 기존 것이 유지됩니다.
