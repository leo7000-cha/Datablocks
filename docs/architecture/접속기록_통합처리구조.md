## 1. 설계 원칙 (V3)

**단일 논리 엔드포인트 — `XAUDIT_DB`**

- 모든 접속기록 I/O (INSERT/SELECT/이상탐지) 가 `XAUDIT_DB` 단일 DataSource 경유
- 물리적 저장소는 `TBL_PIIDATABASE.db='XAUDIT_DB'` 엔트리가 결정
  - 기본: DLM 내부 DB (최초 기동 시 Primary DS 정보로 자동 등록)
  - 변경: UI 에서 해당 엔트리 편집 → 고객사 별도 DB 로 전환
- **모드 개념 없음** (이전 `XAUDIT_STORAGE_MODE=COTDL|EXTERNAL` 제거)
- 업계 Best Practice 패턴:
  - Spring `AbstractRoutingDataSource`
  - AWS CloudTrail (default bucket + redirect)
  - IBM Guardium (Collector + Repository)
  - Oracle AVDF (Repository DB 별도 지정)

## 3. 수집 경로별 수집 대상과 핵심 컬럼

| collect_type | 수집기 | 수집 대상 | 핵심 Master 컬럼 | Sidecar 컬럼 |
|---|---|---|---|---|
| **DB_AUDIT** | `AccessLogCollectorImpl` 배치 | Oracle Unified Audit / MariaDB server_audit | user_account, target_db/table, action_type | sql_text, search_condition |
| **DB_DAC** | `AccessLogCollectorImpl` 배치 | DBSAFER/PSM DAM 장비 로그 | 동일 | sql_text |
| **WAS_AGENT** | `/api/agent/logs` REST | dlm-agent BCI | + session_id | sql_text, target_columns |
| **WAS_SDK** | `/api/xaudit/events` REST | 고객사 처리계 SDK | + req_id, service_name, menu_id, uri, http_*, duration_ms | sql_text(풀), bind_params, full_uri, user_agent |

→ 모두 **동일한** `TBL_ACCESS_LOG` + `TBL_ACCESS_LOG_DETAIL` 에 `collect_type` 으로 구분 저장.

## 4. 부트스트랩 — XAUDIT_DB 자동 등록

최초 기동 시 `XauditDataSourceHolder.@PostConstruct` 가 수행:

```
1. TBL_PIIDATABASE.read('XAUDIT_DB') 조회
2. 엔트리 없음 → Primary DataSource (SPRING_DATASOURCE_URL/USER/PASSWORD) 파싱
3. JDBC URL 정규식 분해: dbtype / host / port / id / id_type
4. AES256Util.encrypt(pwd)
5. INSERT TBL_PIIDATABASE
     db='XAUDIT_DB', system='XAUDIT', env='PRODUCTION',
     dbtype=자동감지(MARIADB/ORACLE/...), hostname/port/id/dbuser/pwd=Primary 값,
     comments='Auto-registered from Primary DataSource'
6. ConnectionProvider.getDataSource(...) 로 전용 Hikari 생성
7. 첫 getConnection().isValid() 검증
```

**즉 아무 설정 없이 DLM 첫 기동만 해도 XAUDIT_DB = DLM 내부 DB 로 자동 동작.**
고객사 별도 DB 로 전환하려면 UI 에서 엔트리 편집 후 재기동.

## 5. 고객사 제공 DB 로 전환 절차 (간소화 V3)

### Step 1 — 고객사 감사 DB 에 스키마 배포

단 2 개 테이블만 생성:
```bash
# MariaDB/MySQL
mariadb -h <host> -u XAUDIT -p <DB명> \
    < deploy/xaudit-sdk-kit/dlm-server/database/XAUDIT_SCHEMA_MARIADB.sql

# Oracle/Tibero
sqlplus XAUDIT/<pwd>@<host>:1521/<svc> \
    @deploy/xaudit-sdk-kit/dlm-server/database/XAUDIT_SCHEMA_ORACLE.sql
```
생성되는 것:
- `TBL_ACCESS_LOG` (Master, 월 파티션)
- `TBL_ACCESS_LOG_DETAIL` (Sidecar, 월 파티션)
- `V_ACCESS_LOG_UNIFIED` (조회 뷰)

### Step 2 — DLM UI 에서 XAUDIT_DB 엔트리 편집
`/pii/database/list` → `XAUDIT_DB` 검색 → 편집:
- hostname / port / dbuser / pwd 를 고객사 DB 값으로 변경
- dbtype 도 고객사 DBMS 로 (ORACLE/TIBERO/MARIADB 등)

또는 직접 UPDATE:
```sql
UPDATE COTDL.TBL_PIIDATABASE
   SET dbtype='ORACLE', hostname='10.x.x.x', port='1521',
       id_type='SERVICE', id='ORCL', dbuser='XAUDIT',
       pwd='<AES256_ENCRYPTED_BASE64>'
 WHERE db='XAUDIT_DB';
```

### Step 3 — DLM 재기동
```bash
docker compose up -d dlm
```
기동 로그에 다음이 찍히면 성공:
```
[X-Audit] XAUDIT_DB DataSource ready: dbtype=ORACLE host=10.x.x.x:1521 user=XAUDIT poolSize=4
```

### Step 4 — smoke-test
```bash
bash deploy/xaudit-sdk-kit/customer/scripts/smoke-test.sh http://<DLM_host>:8080
# → {"inserted":2,"success":true,"received":2}

# 고객사 DB 에서 확인
SELECT COUNT(*) FROM TBL_ACCESS_LOG WHERE collect_type='WAS_SDK';
```

## 6. 실제 데이터가 모이는 흐름 (예시)

고객사 직원 `hong` 이 여신심사 화면을 열어 DB 에 접근:

```
[SDK 내부]
  XauditAccessFilter — ThreadLocal 에 Context (reqId/userId/menu/uri)
  XauditMybatisInterceptor — SQL+bind 가로채기
  BlockingQueue(10K) → 3초/100건 배치 → gzip JSON POST

[DLM 수신]
  XauditEventController.receiveEvents()
    ↓
  XauditEventService.receiveBatch()
    → XauditEventVO 3건 → AccessLogVO 3건 변환
    ↓
  AccessLogService.registerAccessLogBatch()
    ↓
  XauditJdbcWriter.insertUnified()
    1. SELECT last hash_value FROM TBL_ACCESS_LOG ORDER BY log_id DESC LIMIT 1
    2. for each: prev_hash 세팅 + SHA256(userAccount+accessTime+actionType+targetTable+prevHash)
    3. INSERT TBL_ACCESS_LOG (Master, logId 회수)
    4. if vo.hasDetail(): INSERT TBL_ACCESS_LOG_DETAIL (Sidecar)
    5. commit
```

**저장 결과 (V_ACCESS_LOG_UNIFIED 뷰로 한번에 조회 가능)**:
```
log_id=101 HTTP_ACCESS  req_id=abc  user=hong  uri=/loan/review   duration=450ms
log_id=102 SELECT       req_id=abc  user=hong  target_table=CUST  duration=80ms  sql_text=SELECT *...  bind=[CUST_NO=...]
log_id=103 SELECT       req_id=abc  user=hong  target_table=LOAN  duration=50ms  sql_text=SELECT *...  bind=[CUST_NO=...]

해시체인:
  log_id=101 prev=A hash=B
  log_id=102 prev=B hash=C
  log_id=103 prev=C hash=D
```

## 7. 테이블 역할 분담

```
XAUDIT_DB (default: DLM 자체 / 옵션: 고객사 별도 DB)
  TBL_ACCESS_LOG            Master — 모든 collect_type 통합, Lean (고정 길이)
  TBL_ACCESS_LOG_DETAIL     Sidecar — TEXT/가변 필드 (1:1)
  V_ACCESS_LOG_UNIFIED      뷰 — Master ⟗ Sidecar

COTDL (DLM 내부 전용 — 변경 없음)
  TBL_PIIDATABASE           XAUDIT_DB 엔트리 등록소 (핵심!)
  TBL_ACCESS_LOG_ALERT      이상탐지 결과 (분석 산출물)
  TBL_ACCESS_LOG_ALERT_RULE / _SUPPRESSION / ...
  TBL_ACCESS_LOG_HASH_VERIFY / _ARCHIVE_HISTORY
  TBL_ACCESS_LOG_SOURCE / _CONFIG / _DOWNLOAD / _REPORT
```

**분리 원칙**: "**Raw 는 XAUDIT_DB, 분석/운영 산출물은 COTDL**"
→ XAUDIT_DB 가 DLM 자체여도 물리 DB 1개 (테이블 역할만 분리)
→ 외부 DB 로 전환해도 ALERT/REPORT 기능은 COTDL 에 그대로

## 8. 운영상 중요 특성

| 특성 | 설명 |
|---|---|
| **응답 지연 0ms** | SDK 가 BlockingQueue + daemon thread 분리 |
| **DLM 장애 격리** | 큐 포화 drop + warn. 처리계 WAS 무영향 |
| **단일 해시체인 위변조방지** | SHA-256, 4 경로 공유. 금감원 검사 시 체인 검증 1회로 전체 증명 |
| **월별 파티션** | `RANGE COLUMNS(access_time)`. Archive Scheduler 매월 갱신 |
| **PII 빠른 필터** | `pii_detected_flag` CHAR(1) 인덱스 |
| **법규 충족** | 전자금융 13조9항, 개인정보 안전성확보 8조, 신용정보업 별표3 |
| **UI 기반 DB 전환** | XAUDIT_DB 엔트리만 UI 에서 수정 후 재기동 — .env 재배포 불필요 |

## 10. V2 → V3 변경점 요약

| 영역 | V2 (2026-04-24) | V3 (2026-04-25) |
|---|---|---|
| 설계 사상 | 모드 분기 (COTDL / EXTERNAL) | **단일 논리 엔드포인트 (XAUDIT_DB)** |
| 환경변수 | `XAUDIT_STORAGE_MODE`, `DB_KEY`, `SCHEMA` | 모두 제거. `POOL_SIZE` 만 유지 |
| DLM 내부 저장 | MyBatis (AccessLogMapper.xml) | **JdbcTemplate** (XauditAccessLogReader) + JDBC batch (XauditJdbcWriter) |
| 외부 DB 전환 | .env 수정 + 재배포 | **UI 에서 TBL_PIIDATABASE 엔트리만 수정** |
| 부트스트랩 | 고객사가 `XAUDIT_STORAGE_DB_KEY` 등록 필수 | **Primary DS 정보로 자동 등록** (zero config) |
| AccessLogMapper.xml | TBL_ACCESS_LOG INSERT/SELECT 포함 | Master/Sidecar 쿼리 제거 → 부속 테이블 전용 |
| 이상탐지 · 리포트 통계 | MyBatis (그대로) | 그대로 유지 (Phase 2 Reader 이관 예정 — XAUDIT_DB=DLM 자체인 기본 상태에서는 동작) |
